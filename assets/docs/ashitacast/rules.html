Rule Overview

Most of your swaps in ashitacast will be governed by rules.  These are constructed by referencing built in variables, or your own user-configured variables.  There are two kinds of rules you can use, basic or advanced.  Basic rules are intended for simplistic uses where you will only be comparing 1-2 variables.

 

Basic Rules

A basic rule has 2 parts:

<if ad_name=”Thunder V”>

The first part is the attribute name, which does not use a $ prefix like most other instances of variables.  The attribute value will always be a single variable(or user variable, if prefixed with u_).

The second part is the attribute value, where you specify a comparand.  The attribute value can contain strings, numeric values, or variables(if prefixed with $).  You may use several terms seperated by | or & to require only one option to atch, or all options to match.  You may not use both | and & in the same basic rule’s attribute value.  You may start each term with a <= >= < > or !(not equal) symbol to evaluate it as such.  If none are specified, the term will be evaluated as =.  You may use wildcards(*) in basic rules.

 

Some examples:

<if ad_name=”Thunder IV|Thunder V|Thunder VI”>

This will evaluate to true if the value of ad_name (the spell you’re currently casting) is Thunder IV, Thunder V, or Thunder VI since the rule is split by | symbols.

<if ad_name=”!Thunder VI”>

This will evaluate to true if the value of ad_name is NOT Thunder VI, since it begins with !.

<if ad_name=”Thunder*&!Thunder VI”>

This will be split into 2 parts, and evaluated as such:  ($ad_name=Thunder*) AND ($ad_name!Thunder VI).  It will evaluate to true if the value of ad_name starts with Thunder, but is not Thunder VI.  Note that this is pushing the limits of basic rules; while it will function perfectly fine, the chained symbols are often hard to read and you may want to make something like this an advanced rule.

<if ad_mpcost=”>$p_mp”>

In this example, we introduce a second variable and the > operator.  So, we resolve it to: ($ad_mpcost > $p_mp).  If the spell costs more MP than we have, the rule is met(likely to allow us to cancel it or downgrade it!).

<if ad_name=”Convert” p_hp=”<500”>

We can use multiple rules in the same node, and all will have to be met for the node to be true.  So, this rule is met if we’re using convert with below 500 HP.

<if ad_name=”Convert”>

<if mode=”OR” p_hp=”<500 “ p_mp=”>500“>

</if>

</if>

Nested rules are also fine, as they fall into standard flow, as described in XML structure.  In this example, I used the special node mode=”OR” which will require only one of the rules in the node to be met for the node to evaluate as true.

 

Advanced Rules

Advanced rules are used for more complicated comparisons, and can be used in the same way as basic rules.  They will always have an attribute name of advanced.  The entire comparison will be placed as the Attribute Value, and they support additional operands.  Advanced rules can use math and logical operators freely, and should use parentheses to avoid ambiguity.  In an advanced rule, the following operators are evaluated in the following order:
() - Parentheses ensure execution will occur in innermost parentheses first.

^ - Exponents.

*/ - multiplication and division  Note that you cannot use wildcards in advanced rules as they will be interpreted as multiplication.

+- - addition and subtraction

< > <= >= ! = - logical operators, these will convert both sides to a ‘true’ or ‘false’ value

| & - logical operators, these will convert both sides to a ‘true’ or ‘false’ value

 

For an advanced rule to evaluate as true, it must be fully evaluated to a ‘true’ value.  For some examples:

<if advanced=”($p_mp + $p_hp)>1500”>

First, $p_mp and $p_hp will be added together as they are inside parentheses.  Then, the value will be checked against 1500.  If the value is higher than 1500, it will be reduced to the statement ‘true’ and the advanced statement will be met.

<if advanced=”$p_mp+$p_hp>1500|$p_hp>1000”>

Following the order of operations and going left to right, this will be reduced, let’s say you have 1200 HP and 200 MP:

$p_mp+$p_hp>1500|$p_hp>1000

200+1200>1500|1200>1000

1400>1500|1200>1000

False|1200>1000

False|True

True

It is still recommended that you use parentheses for ease of reading; that rule is an extreme eyesore!  Advanced rules use additional string parsing, and should be avoided unless necessary to accomplish your goal.  While you can write any basic rule as an advanced rule, it is rarely a good idea to do so.  This rule would be better expressed as:

<if mode=”OR” p_hp=”>1000” advanced=”($p_mp+$p_hp)>1500”>

Checking Active Buffs

Buffs are checked by using the special variable b_buffid or b_buffname.  These will return the amount of the listed buff you have active.  For example:

<if b_haste=”>=1”> will be true if you have at least 1 copy of the haste buff active.  Note that since it’s using the word haste, it will also count indi-haste or any other buff that displays ‘Haste’ when highlighted.

<if b_33=”>=1”> will also be true if you have at least 1 copy of the haste buff active, but will not count indi-haste, since it is a different buff ID.

You can use these in any way you could use any other variable, as part of a basic or advanced rule.

 

 

Arguments

When writing rules inside a command(input) tag located in the commands section, you also have access to the variables Args Arg1 Arg2 … Arg50.  These variables reference the data sent in the command that triggered the event.

Args will be everything after the command itself.  Leading spaces are removed.

Arg1 is the first argument after the command, Arg2 is the second argument, and so on.

Arguments are seperated by spaces, unless quotes are used to make a multiple word argument.
