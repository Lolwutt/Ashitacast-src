XML Layout

 

<ashitacast> <!-- All files must start with an ashitacast node.  It is the root node. -->

 

            <!--All top-level nodes are strictly optional.  If you don’t need to use one, you may omit it entirely from your XML to conserve space. -->

 

            <settings> <!-- Settings is a top-level node that lets you configure ashitacast’s behavior.-->

 

                        <!-- All setting nodes are optional.  If they are not specified, or an invalid value is used, the default value(listed in white here) will be used. -->

                        <equipbags>0,8,10,11,12</equipbags> <!-- List of container indices you can equip items from.  Should only be changed when on a private server that allows players to equip from non-standard containers. -->

<enablewardrobecheck>true</enablewardrobecheck> <!--If enabled, client will check if wardrobe3/4 are paid before trying to use them.  Should only be false when on a private server that does not properly set values. -->

                        <orderbyindex>false</orderbyindex> <!-- When true, addset will write sets in order of slot index like acv3.  When false, addset will write in order items are showed in equip screen. -->

                        <includeemptyslots>false</includeemptyslots> <!-- When true, addset will include <slot>ignore</slot> nodes for slots that are empty. -->

                        <generatemergedfiles>true</generatemergedfiles> <!-- When true, ashitacast will create a file in config/ashitacast/merged whenever you load a file that uses includes for reference. -->

                        <nakedforpacker>true</nakedforpacker> <!-- When true, ashitacast will remove all gear prior to starting /ac gear.  This ensures that any gear you have equipped can be properly organized. -->

 

                        <!-- Note that all of these values are just timeouts.  If the server sends a packet to indicate action is complete, it will be treated as complete as soon as the packet is received. -->

<!-- Setting them too low will result in your swaps ending before the action completes.  Most users will not need to modify any of these besides snapshot and fast cast. -->

                        <petskilldelay>1500</petskilldelay> <!-- Maximum time(in milliseconds) to wait for a pet weaponskill to complete. -->

                        <weaponskilldelay>1500</weaponskilldelay> <!-- Maximum time(in milliseconds) to wait for a weaponskill to complete. -->

                        <abilitydelay>1500</abilitydelay> <!-- Maximum time(in milliseconds) to wait for a job ability to complete. -->

                        <spelloffset>500</spelloffset> <!-- Additional time(in milliseconds) added to calculated casttime of spells. -->

                        <petspelloffset>500</petspelloffset> <!-- Additional time(in milliseconds) added to calculated casttime of pet spells. -->

                        <rangedoffset>500</rangedoffset> <!-- Additional time(in milliseconds) added to calculated casttime of ranged attacks. -->

                        <itemoffset>500</itemoffset> <!-- Additional time(in milliseconds) added to calculated casttime of items. -->

                        <fastcast>0</fastcast> <!-- Fast cast value to use when calculating casttime of spells. -->

                        <snapshot>0</snapshot> <!-- Snapshot value to use when calculating casttime of ranged attacks. -->

 

            </settings>

 

            <sets> <!-- Sets is a top-level node that is used for listing your equipment sets. -->

 

                        <set name=”Idle” baseset=”Value”> <!-- Set nodes are used to create a new set to be equipped.  You may have any amount of them within your sets node.-->

<!-- name is required.  It is used to reference the set later, and can be anything alphanumeric. -->

<!-- baseset is optional.  If specified, the set will start from the listed baseset and overwrite any pieces listed.-->

 

            <main augpath=”A” augment=”’Fast Cast’+7” augrank=”15” augtrial=”1234” priority=”0” lock=”true”>Bhima</main>

            <!-- EquipSlot Nodes can be named: main, sub, range, ammo, head, neck, ear1, ear2, body, hands, ring1, ring2, back, waist, legs, or feet. -->

            <!-- Augment Attributes can be used to specify a piece beyond it’s name.  All listed Augment Attributes must match for an item to match. -->

            <!-- Priority Attribute is used to determine the order pieces get equipped, primarily to avoid HP or MP drops.  Priority may be set to any value from -2,147,483,648 to 21.47,483,647.  Higher priority is equipped first. -->

            <!-- Lock Attribute is used to prevent a piece being overwritten by later equip nodes.  It only applies to the current top-level node, so for example a lock set in precast would not affect midcast swaps. -->

            <!-- Using Lock Attributes is typically indicative of bad design.  Consider if they are really the best way to accomplish your goal. -->

 

                        </set>

                       

                        <packer> <!-- Packer node is used to list equipment and items that you would like the Packer plugin to retrieve when gearing you using this XML.  Ashitacast itself does not view this node or any subnodes.  You may only have one Packer node, and it must be inside your sets node -->

 

                                    <item augpath=”A” augment=”’Fast Cast’+7” augrank=”15” augtrial=”1234” quantity=”3”>Bhima</item>

                                    <!-- Item Nodes each list an item for packer to retrieve. You may use as many as you want within the packer node. -->

                                    <!-- Augment Attributes are treated the same as in EquipSlot Nodes. -->

                                    <!-- Quantity Attributes allow you to request packer get more than one of an item.  You may specify an integer from 0 to 2,147,483,647 or the string all. -->

 

                        </packer>

            </sets>

 

            <commands> <!-- Commands is a top-level node that is used to react to typed commands while your XML is loaded. -->.

 

                        <command input=”/testcommand”> <!--Command nodes, when placed in the top level of the commands node, are used to define commands.  Anywhere else in the XML, they are used to execute commands. -->

                        <!-- Input Attributes determine the command that a command node will react to.-->

                                   

                                    <!-- The contents of a command node within the top level of the commands node are processed as flow when the matching command is sent. -->

<command>/echo Hello!</command> <!-- Note that command nodes used inside flow are sent to the game and are not the same thing as command nodes in the top level of the commands node. -->

 

                        </command>

                        <!-- This command would send ‘/echo Hello!’ to the game whenever you type /testcommand. -->

                       

            </commands>

 

            <load> <!-- Load is a top-level node.  It is evaluated as flow when your XML is first loaded.  You should set variables’ default values inside load. -->

            </load>

 

            <unload> <!-- Unload is a top-level node.  It is evaluated as flow when your XML is unloaded. -->

            </unload>

           

            <default>> <!-- Default is a top-level node.  It is evaluated as flow every packet cycle(roughly 350-400ms) when you are not currently performing an action or pet action. -->

            </default>

           

            <preranged> <!-- Preranged is a top-level node.  It is evaluated as flow when you perform a ranged attack.  Equip swaps listed here will occur before the ranged attack begins and affect the execution time(use snapshot and rapid shot). -->

            </preranged>

           

            <midranged> <!-- Midranged is a top-level node.  It is evaluated as flow when you perform a ranged attack. Equip swaps listed here will occur before the ranged attack ends and affect the damage and effects. -->

            </midranged>

 

            <precast> <!-- Precast is a top-level node.  It is evaluated as flow when you perform a spell.  Equip swaps listed here will occur before the spell begins and affect it’s cast time(use fast cast and quick magic). -->

            </precast>

 

            <midcast>  <!-- Midcast is a top-level node.  It is evaluated as flow when your perform a spell.  Equip swaps listed here will occur before the spell ends and affect the damage, effects, and recast time. -->

            </midcast>

 

            <ability> <!-- Ability is a top-level node.  It is evaluated as flow when you perform a job ability.  Equip swaps listed here will occur before the ability does and affect damage, effects, and recast time. -->

            </ability>

 

            <weaponskill> <!-- Weaponskill is a top-level node.  It is evaluated as flow when you perform a weaponskill.  Equip swaps listed here will occur before the weaponskill does and affect damage, effects, and TP return. -->

            </weaponskill>

 

            <item> <!-- Item is a top-level node.  It is evaluated as flow when you use an item.  Equip swaps listed here will occur before the item usage finishes. -->

            </item>

 

            <petskill> <!-- Petskill is a top-level node.  It is evaluated as flow when your pet uses a TP move.  Equip swaps will occur as soon as your client is told the pet is performing the TP move.  This may not be fast enough to affect the TP move for some jobs and TP moves that are near-instant.  Keep that in mind when using it. -->

            </petskill>

 

            <petspell> <!-- Petspell is a top-level node.  It is evaluated as flow when your pet casts a spell.  Equip swaps will occur as soon as your client is told the pet has started casting.  This may not be fast enough to affect the spell for some jobs and spells that are near-instant.  Keep that in mind when using it. -->

            </petspell>

</ashitacast>

 

Flow Tags

Unless otherwise specified, these tags may be used in any section of flow and will achieve the same result.

 

<if>, <elseif>, <else>

These tags are used to control flow.  Attributes are covered thoroughly in rules.html.  Some examples of usage are found at the end of this file.

 

<equip set=”SetName” priority=”100” lock=”true”>content</equip>

Equip tags are used to set equipment for the current top-level node.  They can be used in default, preranged, midranged, precast, midcast, ability, weaponskill, item, petskill, and petspell.

They may contain EquipSlot nodes, as described in sets.  If a Set Attribute is present, and the set contains an equipment slot that is also listed as an EquipSlot node, the EquipSlot node will be used for that slot.

If multiple equip commands are executed effecting the same slot, in the same top-level node, the latest equip will take priority.

However, once the Lock Attribute has been set to true for an equipment slot, that equipment slot can only be changed by actions that also have the Lock Attribute set to true until the end of the top-level node.

If the Priority Attribute or Lock Attribute are set for both the equip node itself, and the EquipSlot node(whether part of the set, or specified in the equip tag’s body), the value in the EquipSlot node will be used.

Set Attribute is optional, and determines a set to equip.

Priority Attribute is optional, and sets a default priority for any items that do not have their own Priority Attribute being processed in this tag.

Lock Attribute is optional, and sets a default lock status for any items that do not have their own Lock Attribute being processed in this tag.

 

<command mode=”-1” delay=”1000”>content</command>

Command tags will send a command to the game client, as if you’d typed it.  While they are not self-closing, they may not contain any other nodes.

Content is required, this is where you specify the command to be sent.  Any variables listed in Content will be resolved.

 

Mode Attribute is optional, and will change the command’s mode when it is sent to the client.  Commonly used modes are:

-1 - Parse.  Will be seen by ashita, acts identical to if you’d typed the command.

0 - Menu.  Will not be seen by Ashita, but allows you to use monster IDs.

1 - Typed.  Will not be seen by Ashita, otherwise acts the same as if you’d typed it.

If you do not specify the Mode Attribute, mode will be -1.

 

Delay Attribute is optional, and will introduce a wait(in milliseconds) before the command is sent out.

Note that if you use the Delay Attribute, a thread will be spawned.  This is critical information for advanced users, but should not affect typical use significantly.

 

<echo mode=”0” delay=”1000” color=”1”>content</echo>

Echo tags are similar to command tags, but instead of sending a command, they print text to your chat log.  While they are not self-closing, they may not contain any other nodes.

Content is required, this is where you specify the text to be printed.  Any variables listed in Content will be resolved.

 

Mode Attribute is optional, and will change the text mode when it is sent to the client.  Commonly used modes are:

0 - White, doesn’t appear in any subwindows.

1 - Say(by you)

2 - Shout(by you)

4 - Tell(by you)

5 - Party Chat(by you)

6 - Linkshell Chat(by you)

7 - Emote (by you)

9 - Say(by others)

10 - Shout(by others)

12 - Tell(by others)

13 - Party Chat(by others)

14 - Linkshell Chat(by others)

15 - Emote (by others)

If you do not specify the Mode Attribute, mode will be 0.

Text will appear in the correct tabs, according to the mode it was inserted.  It will also change color, if the color attribute is not specified.

 

Delay Attribute is optional, and will introduce a wait(in milliseconds) before the command is sent out.

Note that if you use the Delay Attribute, a thread will be spawned.  This is critical information for advanced users, but should not affect typical use significantly.

 

Color Attribute is optional, and will change the text color.  It can accept values from 0 to 511.

 

<setvar name=”name” value=”value” />

Setvar tags are used to set a variable.  Note that they are typically self-closing(indicated by ending with /> and not needing a second </tag>), as they do not use content and cannot contain any other nodes.

Name Attribute is required, and is the name of the variable you will be setting.

Value Attribute is required, and is the value you will be setting the variable to.  Note that you may reference other variables and use math in the Value Attribute, in the same manner as the advanced tag described in Variables.html.

 

<clearvar name=”name” all=”true” />

Clearvar tags are used to erase a variable.  Note that they are typically self-closing and cannot contain any other nodes.

Name Attribute is required, and is the name of the variable you will be erasing.

All Attribute is optional, if it is specified with the value of “true”, all variables will be erased.

 

<change id=”id” name=”name” />

Change tags can only be used in the precast, ability, and weaponskill sections.  They are used to change to a different action of the same category before it reaches server.  They are typically self-closing and cannot contain any other nodes.

ID Attribute and Name Attribute both refer to the action you wish to change to.  You must specify at least one, if both are specified then the ID Attribute will take precedence.

 

<gearlock length=”3000” />

Gearlock tags stop all equipment swaps for a period of time.  If you’ve already used equip tags in the same top-level node, they will still go through.  They are typically self-closing and cannot contain any other nodes.

Length Attribute is required, and specifies the duration of the lock in milliseconds.

 

<registerbuff id=”id” name=”name” active=”true|false” />

Registerbuff tags are used to temporarily treat a buff as active or inactive for up to 2 seconds.  They are typically self-closing and cannot contain any other nodes.

These are best used when abilities are used in rapid succession, faster than the server can acknowledge and tell you the buff was gained.

One example would be using Sneak Attack > Trick Attack > Weaponskill.  It is likely the trick attack buff will not yet be active on your screen when you hit weaponskill, so any checks in your weaponskill section will not realize it is up.

You would address this by using registerbuff in the ability section, when you use trick attack.  By giving a 2 second override, the weaponskill will be able to acknowledge that trick attack is active, and the swap will be correct.

ID Attribute and Name Attribute both refer to the buff you wish to register.  You must specify at least one, if both are specified then the ID Attribute will take precedence.

Active Attribute is optional, and will default to true.  If set to false, it will result in the buff showing as inactive.

 

<clearbuff id=”id” name=”name” />

Clearbuff removes a buff from the registry prior to the 2 seconds elapsing.  They are typically self-closing and cannot contain any other nodes.

ID Attribute and Name Attribute both refer to the buff you wish to clear.  You must specify at least one, if both are specified then the ID Attribute will take precedence.

 

<override />

Override tags are used in the petskill and petspell sections.  They have no attributes, are typically self-closing, and cannot contain any other nodes.

If an override tag is hit, no player actions will cause equipment swaps until the pet action associated with that section completes.

Note that this means there are 3 distinct outcomes when a petskill or petspell happens:

·        No matching section exists, or a cancel tag is hit: Nothing happens and swaps continue as normal.

·        A matching section exists, and neither cancel nor override tag is hit: The equipment swap will override default, but if the player initiates an action on their own their swap will still occur.

·        A matching section exists, and override tag is hit: The equipment swap will persist regardless of anything the player does until the pet action completes.

 

<cancel />

Cancel tags are used to stop an action from being sent to server.  They have no attributes, are typically self-closing, and cannot contain any other nodes.

In precast, preranged, ability, weaponskill, and item sections, cancel tags will stop the action from occuring as well as any swaps that would have occurred.  The rest of the section will still parse, so other tags may change variables or create output.

In petskill and petspell sections, cancel tags will not stop the action itself, but will result in ashitacast ignoring it.  This is useful for actions that are too quick to merit swaps, it will save you the swap out of idle gear and back.

In any other section, cancel tags will do nothing.

 

<return />

Return tags act as if the end of the current top-level node was reached, and end processing when touched.  They have no attributes, are typically self-closing, and cannot contain any other nodes.

Return tags do not stop the action, cancel any already processed swaps, or do anything of the sort.  If paired with a cancel tag, ensure the cancel tag is first.

 

Flow

As you can see, many parts of Ashitacast XMLs are simply listed as flow.  This means the sections are very flexible and you can achieve your intended results many different ways.  To start, you’ll need to understand how to use if, elseif, and else tags.  For example usage:

 

<precast>

<if ad_name=”Thunder IV”> <!-- If tags are conditional checks.  If all attributes are met, everything inside the if tag will be evaluated.  If not, processing will skip to after the end of the if tag. -->

                         <command>/echo We’re currently casting Thunder IV!</command>

</if>

<elseif ad_name=”Thunder”> <!--  tags function the same as if tags, but with the condition that they will only be met if the prior if or elseif tag was evaluated and it’s attributes were not met. -->

                        <command>/echo We’re currently casting a Thunder spell besides Thunder IV!</command>

</elseif>

<elseif p_mp=”>1000”> <!-- You can chain as many elseifs as you would like after one if. -->

                        <command>/echo We’re currently casting a non-Thunder spell and we have more than 1000 MP!</command>

</elseif>

<else> <!-- Else tags will be considered met if the prior if or elseif tag was evaluated and it was not met. -->

            <command>/echo We’re currently casting a non-Thunder spell and we have less than or equal to 1000 MP!</command>

</else>

</precast>

 

So, to recap:

·        An elseif statement must always immediately follow the end of an if or elseif statement.

·        An else statement must always immediately follow the end of an if or elseif statement.

This means that a given set of statements will contain one <if> statement, followed by any amount of <elseif> statements(can be zero), followed by an optional else statement.  A set of statements in this configuration will only ever enter one of the results.  Another example:

 

<midcast>

            <if ad_type=”ElementalMagic”>

                        <equip set=”Nuke” />

                        <if ad_name=”Impact”>

                                    <equip>

                                                <head>displaced</head>

<body>Twilight Cloak</body>

                                    </equip>

                        </if>

            </if>

</midcast>

 

To achieve the desired results, you may nest multiple if statements.  Note that no elseif or else statements are required to achieve the effect here.  In that example, if the spell being cast is elemental magic, we equip the nuke set.  Further, if it is also named impact, we overwrite our head and body slot while keeping the rest of the items from the Nuke set.

 

<default>

            <if p_status=”engaged”><equip set=”MyTpSet” /></if>

            <else><equip set=”MyIdleSet” /></else>

            <if p_area=”*Remnants*”><equip><ring1>Balrahn’s Ring</ring1></equip></if>

</default>

 

Adjacent unrelated if statements are fine.  Note here we have two things happening.  First:

·        If you’re engaged, the set named ‘MyTpSet’ is equipped.

·        Otherwise, the set named ‘MyIdleSet’ is equipped.

Second:

·        If you are in a zone with Remnants in the name, your ring1 is overwritten with Balrahn’s Ring.

 

Pay careful attention to the design of your statements to achieve the intended results.  Notice also that in this case I did not indent the contents of tags; this is because XML does not care about your spacing or indentation.  It is recommended that if an <if>, <elseif>, or <else> statement will not terminate in one line without becoming unreasonably long, you create a new line and indent further as in my earlier example to make a visual record of what’s happening.
